## 1. 概述与测试架构

### 1.1 测试设计目标
本规格文档旨在提供一个全方位、多维度、多层级的软件测试用例集，全面覆盖 麦克斯韦妖 重复文件扫描与清理工具的所有功能与非功能特性。重点验证以下系统难点与核心逻辑：
1. **两级哈希算法的精确性**：杜绝误删，保证性能。
2. **多级目录树形继承算法**：保证复杂目录结构下的动作标记准确继承。
3. **安全红线机制**：绝对不允许多个重复文件被全部删除（即 keep 数量不能为 0），整个分类批处理按钮必须与该状态实时精准联动。
4. **手动覆盖与分类不重排的稳定性**：保证良好的用户交互体验。
5. **复杂多场景的文件系统级清理操作**：包括跨物理盘、群晖 NAS 专有废纸篓、写保护及只读异常。

---

### 1.2 测试架构与隔离策略

- **后端单元/集成测试**：
  - **框架**：`pytest` + `pytest-asyncio`
  - **数据库隔离**：利用 SQLite `:memory:`（内存数据库），在每个测试用例生命周期的 `setup` 阶段初始化表结构，`teardown` 阶段自动销毁。
  - **文件系统隔离**：利用 `pytest` 的 `tmp_path` fixture 或 Python 的 `tempfile.TemporaryDirectory` 创建沙盒目录（以下统称 `tmp_fs`），以此模拟真实文件系统。
  - **单例重置**：每个测试执行前重置扫描、分类及移动引擎的全局状态单例。
- **前端状态/组件测试**：
  - **框架**：`vitest` + `@testing-library/react` + `jsdom`
  - **状态重置**：每个测试用例开始前重置 Zustand Store。

## 2. 扫描引擎测试 (SE) — 共 14 案

### SE-01：两级哈希 — 完全相同文件
- **场景描述**：创建两个完全相同的文件，验证初筛与精筛的连贯完整流程。
- **前置条件**：在 `tmp_fs` 隔离区下创建两个文件 `dir1/fileA.txt` 和 `dir2/fileB.txt`，内容均为 `"Hello World"`。
- **操作步骤**：
  1. 对两文件计算头部 64KB xxHash64 摘要。
  2. 验证阶段 1 按 `(xxHash, size)` 将两者放入同一碰撞候选组。
  3. 对该候选组的所有文件流式计算全量 SHA-256。
  4. 验证阶段 2 按 `SHA-256` 最终归为同一重复组。
- **预期结果**：两个文件通过两级哈希最终成功确认为重复组，且写入 `scan_files` 库中。
- **实现状态**：✅ 已实现（对应 `test_se01_identical_files`）

### SE-02：两级哈希 — 头部相同但内容不同
- **场景描述**：两个文件头部（前64KB）相同，但尾部不同，测试精筛过滤能力。
- **前置条件**：创建文件 A（大小 96KB，前 64KB 为随机字符，后 32KB 填充 `0x00`）；创建文件 B（大小 96KB，前 64KB 相同，后 32KB 填充 `0xFF`）。
- **操作步骤**：
  1. 执行扫描。
  2. 验证两文件在阶段 1 被归于同一候选组。
  3. 验证在阶段 2 进行 SHA-256 全量计算后，两者被成功排除，不合并。
- **预期结果**：两文件不会被分入最终的重复组，`scan_files` 库无记录。
- **实现状态**：✅ 已实现（对应 `test_se02_header_same_content_diff`）

### SE-03：两级哈希 — 大小不同快速排除
- **场景描述**：即使两个文件内容有交集，但由于大小不同，在阶段 1 就应当被快速排除，不参与任何全量计算。
- **前置条件**：创建文件 A（内容 `"hello"`），创建文件 B（内容 `"hello world"`）。
- **操作步骤**：
  1. 启动扫描。
  2. 验证阶段 1 计算快速摘要。
  3. 检查阶段 1 输出的碰撞组。
- **预期结果**：因为 size 不同，阶段 1 产生的 `(xxHash, size)` 组合不同，两个文件不被归为候选，不进入阶段 2。
- **实现状态**：✅ 已实现（对应 `test_se03_diff_size`）

### SE-04：超大文件头部哈希性能
- **场景描述**：大文件只读 64KB 进行快速哈希，确保不产生巨大的 I/O 与时间消耗。
- **前置条件**：创建 1 个 2GB 的虚拟大文件（通过 sparse file 或 mock 方式）。
- **操作步骤**：
  1. 调用 `get_xxhash64_64kb()` 接口。
  2. 统计计算头部哈希所用时间。
- **预期结果**：仅读取前 64KB 计算头部 xxHash，计算所用时间 `< 100ms`。
- **实现状态**：✅ 已实现（对应 `test_se04_large_file_head_hash`）

### SE-05：扫描任务安全取消
- **场景描述**：扫描正在高速运行时，用户发起取消请求，系统应安全且立即退出。
- **前置条件**：在 `tmp_fs` 中创建 200 个模拟文件，启动扫描。

- **预期结果**：正在计算哈希的文件计算完毕后，引擎停止后续遍历，不持久化任何中间状态与结果。
- **实现状态**：✅ 已实现（对应 `test_se05_scan_cancel`）

### SE-06：排除目录过滤
- **场景描述**：验证设置排除路径后，特定路径下的目录不被扫描。
- **前置条件**：创建 `photos/`（包含 3 个图片文件），创建 `@eaDir/`（包含群晖缩略图）。设置排除路径含 `@eaDir`。

### SE-07：空文件（0字节）跳过

- **场景描述**：广播循环中，如果有一个旧的连接极其卡顿、网络不通，不能导致向其他健康连接的广播卡死，也不能拖慢整个服务协程。
